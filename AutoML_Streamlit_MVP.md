# AutoML 플랫폼 요구사항 명세서 초안 (Streamlit 기반 / 바이브코딩용)

**문서 버전:** 0.2  
**대상:** MVP 개발용  
**목적:** Streamlit 기반의 빠른 프로토타이핑 방식으로, 정형 데이터 업로드부터 AutoML 학습, 성능 비교, 모델 저장, 예측 수행까지 한 번에 가능한 최소 기능 제품을 정의한다.

---

## 1. 문서 작성 방향

본 문서는 기존 AutoML MVP 요구사항 초안을 바탕으로, 프론트엔드 구성을 일반 웹 프레임워크가 아닌 **Streamlit 기반 UI**로 변경하여 다시 정리한 문서이다.

이번 문서는 특히 다음 목적에 맞춰 작성한다.

- 바이브코딩에 바로 투입 가능한 수준의 구조 제공
- 화면 중심이 아닌 기능 흐름 중심의 빠른 구현 기준 제공
- Python 단일 기술 스택 중심으로 MVP 범위를 단순화
- 추후 FastAPI, Celery, 외부 인증, API 배포 등으로 확장 가능한 구조 유지

---

## 2. 프로젝트 개요

### 2.1 배경

현업 사용자 또는 데이터 분석 초급 사용자가 머신러닝 코드를 직접 작성하지 않고도, 데이터 업로드부터 모델 학습, 결과 비교, 예측 수행까지 한 화면 흐름 안에서 처리할 수 있는 AutoML 도구가 필요하다.

기존 웹 프론트엔드 방식은 개발 분량이 커질 수 있으므로, MVP 단계에서는 Streamlit을 사용하여 화면 개발과 기능 연결을 빠르게 구현한다.

### 2.2 목표

MVP 단계에서는 아래 목표를 달성한다.

- 사용자가 CSV 또는 Excel 데이터를 업로드할 수 있다.
- 사용자가 예측 대상 컬럼과 문제 유형을 선택할 수 있다.
- 시스템이 자동 전처리와 다중 모델 학습을 수행할 수 있다.
- 모델별 성능을 비교하여 베스트 모델을 추천할 수 있다.
- 저장된 모델로 단건 예측과 파일 예측을 수행할 수 있다.
- Streamlit UI에서 전 과정을 끊김 없이 수행할 수 있다.

### 2.3 한 줄 정의

이 MVP는 다음과 같이 정의한다.

**“정형 데이터를 업로드하면 Streamlit 화면에서 분류/회귀 AutoML을 실행하고, 성능 비교 후 저장된 모델로 바로 예측까지 수행할 수 있는 Python 중심 최소 기능 제품”**

## 3. MVP 범위

## 4. 사용자 정의

### 4.1 일반 사용자

- 프로젝트 생성
- 데이터 업로드
- 학습 실행
- 결과 확인
- 저장 모델 예측

### 4.2 관리자

- 프로젝트 및 학습 현황 조회
- 시스템 로그 조회
- 사용자/실행 현황 모니터링

### 4.3 MVP 운영 전제

바이브코딩 기반 초기 버전에서는 다음 중 하나를 선택할 수 있다.

- **단일 사용자 내부 도구 모드**: 로그인 없이 사용
- **간단 로그인 모드**: 아이디/비밀번호 기반 최소 인증

권장안은 1차 MVP에서는 단일 사용자 또는 단순 인증으로 시작하고, 이후 다중 사용자 인증을 붙이는 방식이다.

## 5. 핵심 사용자 시나리오

### 5.1 프로젝트 생성 및 학습

1. 사용자가 Streamlit 앱에 접속한다.
2. 프로젝트를 생성하거나 기존 프로젝트를 선택한다.
3. CSV/XLSX 파일을 업로드한다.
4. 데이터 미리보기와 컬럼 정보를 확인한다.
5. 문제 유형과 타깃 컬럼을 선택한다.
6. 제외 컬럼과 테스트 비율 등 기본 옵션을 설정한다.
7. AutoML 실행 버튼을 누른다.
8. 시스템은 전처리 및 여러 알고리즘 학습을 수행한다.
9. 시스템은 모델별 성능 비교 결과를 화면에 표시한다.
10. 사용자는 추천 모델 또는 원하는 모델을 저장한다.

### 5.2 예측 수행

1. 사용자가 저장된 모델을 선택한다.
2. 단건 입력 또는 파일 업로드 방식 중 하나를 선택한다.
3. 입력값을 넣고 예측을 실행한다.
4. 시스템은 예측 결과를 표 또는 차트 형태로 표시한다.
5. 사용자는 결과를 CSV/XLSX로 다운로드한다.

### 5.3 이력 조회

1. 사용자가 프로젝트 상세 화면으로 이동한다.
2. 과거 학습 이력, 모델 목록, 예측 이력을 확인한다.
3. 사용자는 기존 모델을 재사용하거나 비교 결과를 다시 본다.

## 6.6 자동 전처리

### FR-050 자동 전처리 수행
- 시스템은 학습 전 자동 전처리를 수행해야 한다.

### FR-051 결측치 처리
- 수치형 컬럼은 평균/중앙값 등 정책 기반으로 처리해야 한다.
- 범주형 컬럼은 최빈값 또는 별도 값으로 처리해야 한다.

### FR-052 범주형 인코딩
- 범주형 컬럼은 One-Hot Encoding 또는 적절한 인코딩 방식으로 처리해야 한다.

### FR-053 수치형 스케일링
- 필요 시 StandardScaler 또는 MinMaxScaler 적용이 가능해야 한다.

### FR-054 전처리 기록
- 어떤 전처리가 수행되었는지 실행 이력에 남겨야 한다.

### FR-055 전처리 전략 선택 (사용자 제어)
- 사용자는 "고급 전처리" 영역에서 아래 축의 전략을 기본값 외 대안 중에서 선택할 수 있어야 한다.
  - 수치 결측: `median` / `mean` / `most_frequent` / `constant(0)` / `drop_rows`
  - 수치 스케일: `standard` / `minmax` / `robust` / `none`
  - 이상치 처리: `none` / `iqr_clip` / `winsorize(p)`
  - 범주 결측: `most_frequent` / `constant("__missing__")`
  - 범주 인코딩: `onehot` / `ordinal` / `frequency` (고카디널리티 자동 추천)
- 아무것도 변경하지 않으면 MVP 1차 동작(수치: median→standard / 범주: most_frequent→onehot)과 동일해야 한다. **하위호환 필수**.
- 적용된 전략 조합은 `TrainingJob.run_log` 및 모델 아티팩트에 함께 기록되어야 한다 (FR-054 확장).

### FR-056 고급 타입 처리
- 시스템은 `datetime64` 및 문자열 기반 날짜 컬럼을 자동 감지하고, 사용자 선택 시 `year / month / day / weekday / hour / is_weekend` 등의 파생 피처를 생성할 수 있어야 한다.
- 시스템은 `bool` 타입 컬럼을 0/1 수치형으로 정규화해 학습에 활용할 수 있어야 한다 (현재는 분류 후보에서 누락되는 케이스 보완).
- 고카디널리티 범주 컬럼(예: `nunique > 50` 또는 `unique_ratio > 0.3`) 은 자동으로 감지되고, 사용자에게 대안 인코딩(frequency 권장)이 기본 선택으로 제안되어야 한다.

### FR-057 클래스 불균형 대응 (분류 전용)
- 분류 문제에서 사용자는 다음 중 하나를 선택할 수 있어야 한다.
  - `none` (기본)
  - `class_weight="balanced"` (estimator 지원 시)
  - `SMOTE` 리샘플링 (`imbalanced-learn` 의존)
- `SMOTE` 는 **학습 fold 내부**에서만 적용되어야 하며, 평가(test) 세트에는 절대 적용되지 않아야 한다 (데이터 누수 방지).
- 선택한 대응 방식은 `run_log` 에 기록된다.

### FR-058 피처 변환 미리보기
- 학습 실행 **전**, 사용자는 현재 전처리 설정을 반영했을 때의 요약을 확인할 수 있어야 한다.
  - 원본 열 수 → 변환 후 열 수
  - 생성될 파생 피처 목록(출처 컬럼 → 생성 컬럼)
  - 인코딩/스케일링/이상치 처리 방식 요약
- 미리보기는 **실제 fit 없이 메타데이터 기반** 으로 저비용 계산되어야 한다 (NFR-003 범위 내, 5만 행 기준 2초 이내 목표).

## 6.7 AutoML 학습 실행

### FR-060 학습 실행
- 사용자는 Streamlit 화면에서 AutoML 실행 버튼을 눌러 학습을 시작할 수 있어야 한다.

### FR-061 다중 알고리즘 실행
- 시스템은 문제 유형에 맞는 복수 알고리즘을 자동 실행해야 한다.

### FR-062 지원 알고리즘
- 분류: Logistic Regression, Random Forest, Decision Tree, XGBoost 또는 LightGBM
- 회귀: Linear Regression, Random Forest Regressor, Ridge, Lasso, XGBoost Regressor 또는 LightGBM Regressor

### FR-063 데이터 분할
- 시스템은 train/test 또는 train/valid/test 분할을 자동 수행해야 한다.

### FR-064 성능 측정
- 시스템은 알고리즘별 평가지표를 계산해야 한다.

### FR-065 진행 상태 표시
- 사용자는 학습 진행 상태를 확인할 수 있어야 한다.
- 상태 값 예: 대기, 실행 중, 완료, 실패
- Streamlit progress bar 또는 status 영역으로 표시할 수 있어야 한다.

### FR-066 실패 처리
- 학습 실패 시 사용자에게 실패 원인과 점검 포인트를 표시해야 한다.

### FR-067 알고리즘 선택 (사용자 제어)
- 사용자는 학습 실행 전에 지원 알고리즘 목록(FR-062 확장) 중 원하는 부분집합을 체크박스(`st.multiselect`)로 선택할 수 있어야 한다.
- 선택을 변경하지 않으면 "사용 가능한 전체 알고리즘"이 기본값이며, 이 경우 v0.2.0 이전 동작과 완전히 동일해야 한다 (감사 로그 미발생).
- 최소 1개 이상이 선택되어야 학습을 시작할 수 있다. 0개 선택 시 UI 경고 후 학습을 차단해야 한다.
- 부분 선택이 이루어진 경우 `TrainingJob.run_log` 및 감사 로그 `training.algorithms_filtered` 에 선택된 알고리즘 목록이 기록되어야 한다.
- 문제 유형 변경(`classification` ↔ `regression`) 시 해당 task 에서 유효하지 않은 선택 항목은 자동으로 제거되어야 한다.

### FR-068 확장 알고리즘 지원
- 분류 지원: Logistic Regression, Random Forest, Decision Tree, XGBoost, LightGBM, **Hist Gradient Boosting, Extra Trees, Gradient Boosting, K-Nearest Neighbors**, (CatBoost: optional)
- 회귀 지원: Linear Regression, Random Forest Regressor, Ridge, Lasso, XGBoost Regressor, LightGBM Regressor, **Hist Gradient Boosting Regressor, Extra Trees Regressor, Gradient Boosting Regressor, K-Nearest Neighbors Regressor, ElasticNet, Decision Tree Regressor**, (CatBoost Regressor: optional)
- Tier 1 신규 알고리즘은 sklearn 내장 기반이며 필수 의존성이므로 CI/기본 환경에서 항상 사용 가능하다.

### FR-069 선택적 백엔드 (Optional Backend)
- CatBoost 는 선택적 백엔드로 제공되며, `requirements-optional.txt` 에 분리 기재한다 (기본 CI/설치에 포함되지 않는다).
- 선택적 백엔드(XGBoost, LightGBM, CatBoost)는 import 실패 시 학습 대상에서 자동 제외되며, UI 는 `training_service.list_optional_backends()` 로부터 상태와 사용자 행동 가능한 사유(예: "pip install required", "libomp 누락")를 받아 캡션으로 표시해야 한다.
- UI(`pages/*`)는 `ml/registry` 를 직접 import 하지 않고, 반드시 `services/training_service` 의 `list_algorithms(task_type)` / `list_optional_backends()` API 를 경유해야 한다 (레이어 경계).

## 10. 처리 규칙

### 10.1 데이터 업로드 규칙
- 허용 확장자: csv, xlsx
- 헤더 없는 파일은 업로드 불가
- 중복 컬럼명은 오류 처리 또는 자동 보정 처리
- 파일 업로드 즉시 샘플 미리보기와 컬럼 프로파일링을 수행

### 10.2 학습 규칙
- 타깃 컬럼은 반드시 1개 선택해야 한다.
- 타깃 컬럼은 입력 피처에서 제외한다.
- 고유값이 과도하게 많은 식별자 컬럼은 제외 후보로 안내한다.
- 문제 유형에 맞지 않는 타깃 컬럼이면 경고를 표시한다.

### 10.3 모델 선정 규칙
- 분류 기본 기준: F1 또는 Accuracy
- 회귀 기본 기준: RMSE 또는 MAE
- 사용자가 기준 지표를 변경하면 추천 모델 기준도 함께 바뀌어야 한다.

### 10.4 예측 규칙
- 학습 시점 입력 스키마와 예측 입력 스키마가 일치해야 한다.
- 누락 컬럼은 예측 차단 대상이다.
- 추가 컬럼은 경고 후 무시할 수 있다.

- Streamlit은 일반적인 SPA 프론트엔드가 아니라, **화면과 Python 로직을 빠르게 묶는 인터랙티브 UI 계층**으로 보는 것이 맞다.
- 따라서 이 문서에서는 프론트엔드 요구사항을 페이지 디자인 요구보다 **작업 흐름과 위젯 중심 요구사항**으로 정리했다.
- 바이브코딩 시에는 먼저 권장안 A로 시작하고, 이후 필요 시 FastAPI + Worker 구조로 분리하는 방식이 현실적이다.
